= Musik in Schaumburg
:toc: left
:icons: font
:source-highlighter: highlight.js

Landing Page für Orchester im Landkreis Schaumburg, Niedersachsen, Deutschland.

== Überblick

Diese Seite bietet eine Übersicht aller Orchester, Blasorchester, Chöre und Musikgruppen im Landkreis Schaumburg.

*Betreuer:* Benjamin Marwell — link:https://bmarwell.de[bmarwell.de]

== Schnellstart

Abhängigkeiten installieren und die Seite bauen:

[source,bash]
----
bun install
bun run build
----

Vorschau der gebauten Seite im Browser starten:

[source,bash]
----
bun run preview
----

Die gebaute Seite liegt im `dist/`-Verzeichnis.

== Struktur

[source]
----
musik-in-schaumburg.de/
├── orchestras/                 # Orchesterdaten
│   └── <slug>/
│       └── index.yaml          # Orchesterkonfiguration
├── src/main/html/              # Quelldateien
│   ├── index.html              # Startseite (Template)
│   ├── orchestra.html          # Orchesterseite (Template)
│   ├── css/main.css            # Stylesheet
│   └── js/lightbox.js          # Lightbox-Skript
├── scripts/
│   └── build-pipeline.mjs     # Build-Pipeline
├── dist/                       # Gebaute Seite (nicht ins Git)
└── package.json
----

== Orchester hinzufügen

Um ein neues Orchester hinzuzufügen, legt man einen neuen Ordner unter `orchestras/<slug>/` an mit einer `index.yaml`-Datei:

[.note]
====
Stichworte / Tags:

- Es gibt eine zentrale Liste mit auswählbaren Stichworten in `src/main/keywords.yml`.
- In `index.yaml` kann das Feld `tags:` als Liste (oder `keywords:`) angegeben werden, z.B.

[source,yaml]
----
tags:
  - Jugend
  - Kirche
  - Blasmusik
----

- Falls ein Stichwort nicht in der zentralen Liste vorhanden ist, wird ein Warnhinweis beim Build ausgegeben, das Stichwort aber übernommen.
====

[source,yaml]
----
title: "Name des Ensembles"
type: brass-band            # brass-band | choir | posaunenchor | symphony | big-band | ...
slug: mein-ensemble         # URL-Pfad, z.B. /orchester/mein-ensemble/

logo:
  local: "logo.png"         # BEVORZUGT: Datei liegt in orchestras/<slug>/logo.png
  # url: https://...        # Fallback falls kein lokales Logo vorhanden

image:
  local: "photo.jpg"        # BEVORZUGT: Datei liegt in orchestras/<slug>/photo.jpg
  # url: https://...        # Fallback falls kein lokales Bild vorhanden

description: "Kurzbeschreibung des Ensembles."

location: "Stadthagen"      # optional: Probenort / Standort
website: "https://example.com"        # optional
social:
  facebook: "https://www.facebook.com/..."   # optional
  instagram: "https://www.instagram.com/..."  # optional
  youtube: "https://www.youtube.com/..."      # optional
----

=== Bilder und Logos

Bilder und Logos sollten **immer lokal** im Verzeichnis `orchestras/<slug>/` abgelegt und in der YAML-Datei mit `image.local` bzw. `logo.local` referenziert werden.
`image.url` / `logo.url` dienen nur als Fallback, wenn kein lokales Asset verfügbar ist, und erzeugen beim Build eine Warnung.

=== Orchestertypen

[cols="1,2"]
|===
| Typ | Bedeutung

| `brass-band`      | Brass Band
| `symphony`        | Sinfonieorchester / Sinfonisches Blasorchester
| `choir`           | Chor
| `posaunenchor`    | Posaunenchor
| `big-band`        | Big Band
| `school-band`     | Schulkapelle / Schülerorchester
| `chamber`         | Kammerorchester / -ensemble
| `wind-ensemble`   | Bläserensemble
| `strings-ensemble`| Streichensemble
| `other`           | Sonstiges
|===

== Build-Pipeline

Die Build-Pipeline (`scripts/build-pipeline.mjs`) führt folgende Schritte durch:

1. `dist/`-Verzeichnis leeren
2. YAML-Dateien der Orchester einlesen
3. Bilder und Logos herunterladen (falls URLs angegeben)
4. Responsive WebP-Varianten erstellen (400 px, 800 px, 1200 px)
5. HTML-Seiten aus Templates generieren (Startseite + je eine Seite pro Orchester)
6. `sitemap.xml` generieren
7. CSS, JavaScript, `robots.txt` und `.htaccess` kopieren
8. Alle Textdateien vorab komprimieren: Brotli (`.br`), gzip (`.gz`) und zstd (`.zst`)

=== Vorab-Komprimierung

Jede Textdatei in `dist/` (HTML, CSS, JS, XML, TXT, SVG) wird in drei komprimierte
Varianten erstellt. Apache liefert die passende Variante anhand des vom Browser gesendeten
`Accept-Encoding`-Headers aus (Priorität: zstd > brotli > gzip). Falls keine vorab
komprimierte Datei vorhanden ist, greift Apache auf die Echtzeit-Komprimierung via
`mod_brotli` / `mod_deflate` zurück.
